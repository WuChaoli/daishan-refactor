# Log Decorator 功能优化需求文档

## 1. 背景

当前 `log_decorator` 模块在日志输出方面存在以下可用性问题：

1. **参数显示不直观**：日志中显示 `args[0]: 危化品` 而非 `query: 危化品`，无法快速识别参数含义
2. **日志结构扁平**：参数信息未纳入树形结构，层级关系不清晰
3. **性能开销**：默认计算耗时影响高频调用场景的性能
4. **参数冗余**：`enable_mermaid` 参数与 `force_mermaid` 功能重复
5. **日志覆盖风险**：并发场景下入口函数日志文件可能被覆盖

这些问题降低了日志的可读性和实用性，增加了调试成本。

## 2. 目标

通过优化 `log_decorator` 模块，实现以下目标：

- **提升可读性**：参数显示清晰，树形结构完整
- **优化性能**：默认模式下减少不必要的计算开销
- **简化接口**：清理冗余参数，降低使用复杂度
- **保证完整性**：防止并发场景下的日志覆盖

## 3. 功能描述

### 3.1 参数名称显示优化

**功能说明**：
通过函数签名（`inspect.signature`）获取参数名称，在日志中显示为 `参数名: 参数值` 格式。

**实现要求**：
- 位置参数：显示为 `param_name: value`（如 `query: 危化品`）
- 关键字参数：显示为 `param_name: value`（如 `context: None`）
- 默认参数：显示实际传入的值或默认值
- 可变参数：`*args` 显示为 `args: (value1, value2)`，`**kwargs` 显示为 `kwargs: {key: value}`

**示例对比**：
```
# 优化前
args[0]: 危化品
args[1]: None

# 优化后
query: 危化品
context: None
```

### 3.2 参数树形结构

**功能说明**：
将参数信息作为函数节点的子节点，使用树形结构显示，保持层级关系清晰。

**实现要求**：
- 参数节点作为 `【入参】` 和 `【出参】` 的子节点
- 使用树形符号（`├─`、`│`、`└─`）表示层级关系
- 保持与现有日志树结构的一致性

**输出格式**：
```
├─ process_intent - 【开始执行】
│  ├─ 【入参】
│  │  ├─ query: 危化品
│  │  └─ context: None
│  ├─ extract_keywords - 【开始执行】
│  │  ├─ 【入参】
│  │  │  └─ text: 危化品
│  │  └─ 【返回】keywords: ['危化', '化品']
│  └─ 【返回】intent: safety_query
```

### 3.3 耗时计算默认关闭

**功能说明**：
将 `print_duration` 参数默认值改为 `False`，仅在显式指定时才计算和输出函数耗时。

**实现要求**：
- 修改 `log()` 装饰器签名：`def log(print_duration=False, ...)`
- 当 `print_duration=False` 时，跳过 `time.time()` 调用和耗时计算
- 当 `print_duration=True` 时，输出格式保持不变：`【耗时】0.123s`

**使用示例**：
```python
# 默认不计算耗时
@log()
def fast_function():
    pass

# 显式启用耗时计算
@log(print_duration=True)
def slow_function():
    time.sleep(1)
```

### 3.4 删除 enable_mermaid 参数

**功能说明**：
从 `log()` 函数签名中删除 `enable_mermaid` 参数，保留 `force_mermaid` 参数用于控制 Mermaid 图生成。

**实现要求**：
- 从 `decorator.py` 中的 `log()` 函数签名删除 `enable_mermaid` 参数
- 清理所有与 `enable_mermaid` 相关的逻辑判断
- 保留 `force_mermaid` 参数及其功能
- 确保删除后不影响现有 Mermaid 图生成功能

**修改前后对比**：
```python
# 修改前
def log(enable_mermaid=True, force_mermaid=False, ...):
    ...

# 修改后
def log(force_mermaid=False, ...):
    ...
```

### 3.5 入口函数重复防止

**功能说明**：
日志文件名使用 `{func_name}_{timestamp}.log` 格式，防止并发场景下的文件覆盖。

**实现要求**：
- 时间戳格式：`YYYYMMDD_HHMMSS`（如 `20260209_173844`）
- 文件名格式：`{func_name}_{timestamp}.log`（如 `process_intent_20260209_173844.log`）
- 仅对入口函数（调用栈顶层）应用此规则
- 非入口函数继续使用原有命名规则

**示例**：
```
# 并发调用场景
process_intent_20260209_173844.log  # 第一次调用
process_intent_20260209_173845.log  # 第二次调用（1秒后）
process_intent_20260209_173845_001.log  # 第三次调用（同一秒内，添加毫秒后缀）
```

## 4. 验收标准

### 4.1 功能验收

| 优化点 | 验收标准 |
|--------|----------|
| 参数名称显示 | ✓ 日志中显示 `param_name: value` 格式<br>✓ 支持位置参数、关键字参数、默认参数<br>✓ 正确处理 `*args` 和 `**kwargs` |
| 参数树形结构 | ✓ 参数作为 `【入参】`/`【出参】` 的子节点<br>✓ 使用正确的树形符号<br>✓ 层级缩进正确 |
| 耗时计算 | ✓ 默认 `print_duration=False`<br>✓ 不计算耗时时无性能开销<br>✓ 显式启用时正常输出耗时 |
| 删除参数 | ✓ `enable_mermaid` 从函数签名中删除<br>✓ 相关逻辑已清理<br>✓ `force_mermaid` 功能正常 |
| 防止覆盖 | ✓ 文件名包含时间戳<br>✓ 并发调用生成不同文件<br>✓ 同一秒内调用添加毫秒后缀 |

### 4.2 非功能验收

- **性能要求**：默认模式（`print_duration=False`）下，装饰器开销 < 0.5ms
- **兼容性要求**：现有使用 `@log()` 的代码无需修改即可运行
- **代码质量**：符合 Python 开发规范（函数 ≤30行，文件 ≤800行，嵌套 ≤4层）
- **测试覆盖**：单元测试覆盖率 ≥ 80%

### 4.3 测试场景

1. **场景1：基本参数显示**
   - 输入：`func(query="危化品", context=None)`
   - 预期：日志显示 `query: 危化品` 和 `context: None`

2. **场景2：树形结构**
   - 输入：嵌套函数调用（A调用B，B调用C）
   - 预期：参数正确嵌套在各自函数节点下

3. **场景3：性能测试**
   - 输入：高频调用（10000次/秒）
   - 预期：默认模式下无明显性能下降

4. **场景4：并发调用**
   - 输入：同时触发3个入口函数调用
   - 预期：生成3个不同的日志文件，无覆盖

## 6. 风险与依赖

**风险**：
- 参数解析可能影响装饰器性能（需性能测试验证）
- 时间戳精度不足可能导致同一毫秒内的调用仍然冲突（需添加随机后缀）

**依赖**：
- Python 标准库 `inspect` 模块（用于函数签名解析）
- 无新增外部依赖
