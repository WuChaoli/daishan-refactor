# 用户故事 - Log Decorator Optimization

## 概述

本文档从开发者、调试人员、运维人员三个角色的视角，描述 log_decorator 功能优化的使用场景和价值。

---

## 优化点 1：优化参数显示

### 用户故事 1.1 - 开发者视角

**作为**开发者，**我想要**在日志中看到函数参数的名称和值（如 `user_id=123, action="update"`），**以便**快速理解函数调用时传入了哪些参数，无需查看源代码。

**验收条件：**
- [ ] 日志中显示 `param_name=value` 格式
- [ ] 支持位置参数和关键字参数的名称显示
- [ ] 返回值显示变量名或类型信息
- [ ] 复杂对象（如字典、列表）的参数能正确解析并显示

### 用户故事 1.2 - 调试人员视角

**作为**调试人员，**我想要**在日志文件中直接看到参数名称，**以便**在排查问题时快速定位错误参数，而不需要对照代码推断参数位置。

**验收条件：**
- [ ] 日志文件中参数名称清晰可读
- [ ] 参数值格式化输出，避免过长内容影响可读性
- [ ] 支持通过参数名搜索日志
- [ ] 异常场景下参数信息完整记录

---

## 优化点 2：参数纳入树结构

### 用户故事 2.1 - 开发者视角

**作为**开发者，**我想要**参数信息以树形结构展示在函数节点下，**以便**理解函数调用的层级关系和参数传递链路。

**验收条件：**
- [ ] 参数作为子节点显示在函数节点下
- [ ] 保持正确的缩进和层级关系
- [ ] 支持嵌套参数的展开显示
- [ ] 树结构与函数调用栈对应

### 用户故事 2.2 - 调试人员视角

**作为**调试人员，**我想要**通过树形结构快速定位调用链中的参数传递，**以便**追踪数据在多层函数调用中的变化过程。

**验收条件：**
- [ ] 树形结构层级清晰，易于阅读
- [ ] 可以快速识别父子函数的参数关系
- [ ] 支持折叠/展开功能（如果是交互式日志查看器）
- [ ] 参数变化在不同层级中可追溯

---

## 优化点 3：按需计算耗时

### 用户故事 3.1 - 开发者视角

**作为**开发者，**我想要**控制是否计算函数耗时（默认关闭），**以便**在性能敏感的高频调用场景下减少装饰器开销。

**验收条件：**
- [ ] 默认 `enable_timing=False`，不计算耗时
- [ ] 显式传入 `@log(enable_timing=True)` 时才计算耗时
- [ ] 不影响现有日志结构和其他功能
- [ ] 性能测试显示默认模式下开销降低 ≥ 50%

### 用户故事 3.2 - 运维人员视角

**作为**运维人员，**我想要**在需要性能分析时启用耗时统计，**以便**识别系统瓶颈，同时在正常运行时减少日志系统的性能影响。

**验收条件：**
- [ ] 可以通过配置文件全局控制耗时统计
- [ ] 耗时数据格式统一，便于分析工具解析
- [ ] 支持动态开启/关闭耗时统计（无需重启服务）
- [ ] 耗时统计不影响业务逻辑的执行时间

---

## 优化点 4：删除 enable_mermaid 参数

### 用户故事 4.1 - 开发者视角

**作为**开发者，**我想要**简化装饰器接口，移除冗余的 `enable_mermaid` 参数，**以便**减少配置复杂度，降低使用门槛。

**验收条件：**
- [ ] 从 `@log()` 函数签名中移除 `enable_mermaid` 参数
- [ ] 清理相关的条件判断逻辑
- [ ] 确保不影响其他功能的正常运行
- [ ] 现有使用该装饰器的代码无需修改即可运行

### 用户故事 4.2 - 运维人员视角

**作为**运维人员，**我想要**装饰器接口简洁明了，**以便**在配置和维护日志系统时减少理解成本和配置错误。

**验收条件：**
- [ ] 装饰器参数列表清晰，无冗余选项
- [ ] 配置文档更新，移除相关说明
- [ ] 不影响现有日志文件的生成和管理
- [ ] 简化后的接口更易于团队培训和推广

---

## 优化点 5：防止入口函数覆盖

### 用户故事 5.1 - 开发者视角

**作为**开发者，**我想要**保留所有入口函数的日志记录，**以便**在并发场景下追踪完整的调用链，不会因为重复调用而丢失日志。

**验收条件：**
- [ ] 检测重复的入口函数调用
- [ ] 使用唯一标识（如时间戳、UUID、调用ID）区分不同调用
- [ ] 所有调用记录完整保留，不相互覆盖
- [ ] 支持通过唯一标识关联同一次请求的所有日志

### 用户故事 5.2 - 调试人员视角

**作为**调试人员，**我想要**在日志文件中看到所有入口函数的调用记录，**以便**排查并发场景下的问题，确保每次请求的日志都能完整追溯。

**验收条件：**
- [ ] 日志文件中每次入口函数调用都有独立记录
- [ ] 唯一标识清晰可见，便于搜索和过滤
- [ ] 并发调用的日志不会混淆或覆盖
- [ ] 支持按唯一标识导出单次请求的完整日志

### 用户故事 5.3 - 运维人员视角

**作为**运维人员，**我想要**日志系统能够处理高并发场景下的入口函数调用，**以便**确保日志完整性，支持问题追溯和性能分析。

**验收条件：**
- [ ] 高并发场景下日志不丢失、不覆盖
- [ ] 日志文件大小和数量可控，支持自动轮转
- [ ] 唯一标识生成机制高效，不影响性能
- [ ] 支持通过日志分析工具聚合和分析并发请求

## 综合场景

### 场景 1：开发者调试新功能

**背景：**开发者正在开发一个新的数据处理功能，需要记录函数调用日志以便调试。

**步骤：**
1. 在函数上添加 `@log()` 装饰器
2. 调用函数 `process_data(user_id=123, action="update", data={"name": "test"})`
3. 查看日志文件，看到清晰的参数显示：`user_id=123, action="update", data={"name": "test"}`
4. 参数以树形结构展示在函数节点下，层级清晰
5. 默认不计算耗时，装饰器开销极小

**价值：**快速定位问题，提升开发效率。

---

### 场景 2：调试人员排查生产问题

**背景：**生产环境出现间歇性错误，调试人员需要通过日志追踪问题根源。

**步骤：**
1. 打开日志文件，搜索错误发生时间段的日志
2. 通过参数名称快速定位可疑的参数值
3. 通过树形结构追踪调用链，查看参数在不同层级的传递
4. 通过唯一标识过滤出单次请求的完整日志
5. 定位到问题函数和错误参数

**价值：**快速定位问题，减少故障恢复时间。

---

### 场景 3：运维人员性能优化

**背景：**系统响应变慢，运维人员需要分析性能瓶颈。

**步骤：**
1. 通过配置文件全局启用 `enable_timing=True`
2. 收集一段时间的日志数据
3. 使用日志分析工具解析耗时数据
4. 识别出耗时最长的函数和调用路径
5. 优化完成后关闭耗时统计，恢复正常性能

**价值：**精准定位性能瓶颈，优化系统性能。

---

## 总结

本次优化覆盖了 log_decorator 的5个核心痛点，从开发者、调试人员、运维人员三个角色的视角出发，提升了日志的可读性、性能和可维护性。每个优化点都有明确的验收条件，确保功能实现符合用户期望。
