# 需求文档 - Log Decorator Optimization

## 概述

优化 `src/log_decorator` 模块的日志装饰器功能，提升参数可读性、优化性能、改进配置灵活性，并解决入口函数重复覆盖问题。

## 背景

- **问题**：当前 log_decorator 存在以下问题：
  1. 入参和出参显示不直观，难以快速定位参数
  2. 参数未纳入树结构，日志层级不清晰
  3. 默认计算耗时影响性能，应按需启用
  4. enable_mermaid 参数冗余，需要清理
  5. 入口函数可能重复导致日志覆盖
- **业务价值**：提升日志可读性和性能，降低调试成本

## 功能需求

### 核心功能

1. **优化参数显示**
   - 描述：入参和出参需要显示参数名称，而不是仅显示值
   - 用户故事：作为开发者，我想要在日志中看到参数名称，以便快速理解函数调用
   - 验收标准：
     - 日志中显示 `param_name=value` 格式
     - 支持位置参数和关键字参数
     - 返回值显示变量名或类型信息

2. **参数纳入树结构**
   - 描述：将参数信息整合到日志树结构中
   - 用户故事：作为开发者，我想要参数以树形结构展示，以便理解调用层级
   - 验收标准：
     - 参数作为子节点显示在函数节点下
     - 保持缩进和层级关系
     - 支持嵌套参数展开

3. **按需计算耗时**
   - 描述：默认不计算耗时，仅在明确指定时启用
   - 用户故事：作为开发者，我想要控制是否计算耗时，以便在性能敏感场景下减少开销
   - 验收标准：
     - 默认 `enable_timing=False`
     - 显式传入 `enable_timing=True` 时才计算
     - 不影响现有日志结构

4. **删除 enable_mermaid 参数**
   - 描述：移除 log 主函数的 enable_mermaid 入参
   - 用户故事：作为开发者，我想要简化装饰器接口，以便减少配置复杂度
   - 验收标准：
     - 从函数签名中移除该参数
     - 清理相关逻辑
     - 确保不影响其他功能

5. **防止入口函数覆盖**
   - 描述：设计策略防止重复的入口函数日志被覆盖
   - 用户故事：作为开发者，我想要保留所有入口函数的日志，以便追踪完整调用链
   - 验收标准：
     - 检测重复入口函数调用
     - 使用唯一标识（如时间戳、调用ID）区分
     - 保留所有调用记录

### 非功能需求

- **性能要求**：默认模式下装饰器开销 < 1ms
- **兼容性要求**：保持现有日志格式向后兼容
- **可维护性要求**：代码符合 Python 开发规范（函数 ≤30行，文件 ≤800行）

## 用户场景

### 场景 1：查看函数调用参数

1. 开发者调用被装饰的函数 `process_data(user_id=123, action="update")`
2. 系统输出日志显示 `user_id=123, action="update"`
3. 开发者快速定位参数值

### 场景 2：性能敏感场景

1. 开发者在高频调用函数上使用装饰器
2. 系统默认不计算耗时，减少性能开销
3. 需要性能分析时，显式启用 `enable_timing=True`

### 场景 3：并发入口函数调用

1. 多个请求同时触发入口函数
2. 系统为每个调用生成唯一标识
3. 所有调用日志完整保留，不相互覆盖

## 约束条件

- **技术约束**：必须兼容现有 log_decorator 使用方式
- **时间约束**：需在本周内完成优化
- **资源约束**：不引入新的外部依赖

## 成功标准

- 所有 5 个优化点完成实现
- 通过单元测试覆盖率 ≥ 80%
- 现有使用 log_decorator 的代码无需修改即可运行
- 性能测试显示默认模式下开销降低 ≥ 50%
