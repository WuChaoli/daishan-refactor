# 猜你想问功能 - 验收标准

## 功能验收标准

### 1. 接口基本功能

#### 1.1 接口可访问性
- [ ] 接口路径 `POST /api/guess-questions` 可正常访问
- [ ] 接口接受 JSON 格式的请求体
- [ ] 接口返回 JSON 格式的响应

#### 1.2 输入参数验证
- [ ] 接受 `question` 参数（字符串类型）
- [ ] `question` 参数为空时返回错误提示
- [ ] `question` 参数过长（>1000字符）时正常处理或返回错误提示

### 2. 类别1推荐问题

#### 2.1 固定问题返回
- [ ] 当意图类型为1时，返回3个固定的推荐问题
- [ ] 返回的问题数量固定为3个
- [ ] 返回格式符合规范：`[{"guess_your_question": "XXX"}, ...]`

#### 2.2 问题内容
- [ ] 固定问题内容为预设的模板
- [ ] 问题内容清晰易懂，符合业务场景
- [ ] 问题内容可配置修改（代码层面）

### 3. 类别2推荐问题

#### 3.1 问题提取
- [ ] 当意图类型为2时，从 `results` 列表中提取第2-4个结果
- [ ] 正确提取每个结果的 `question` 字段
- [ ] 返回格式符合规范：`[{"guess_your_question": "XXX"}, ...]`

#### 3.2 边界情况处理
- [ ] 当 `results` 列表少于4个元素时，返回实际可用的结果
- [ ] 当 `results` 列表为空时，返回空列表
- [ ] 当 `results` 列表只有1个元素时，返回空列表（因为需要第2-4个）

### 4. 其他类别处理

#### 4.1 类别0或其他
- [ ] 当意图类型为0或其他未定义类型时，返回空列表或默认推荐问题
- [ ] 不会因为未知类型而抛出异常

### 5. 错误处理

#### 5.1 意图识别服务异常
- [ ] 当 `IntentService.process_query()` 调用失败时，捕获异常
- [ ] 返回统一的错误响应格式：`{"code": 1, "message": "错误信息", "data": []}`
- [ ] 错误信息清晰明确，便于排查问题

#### 5.2 数据格式异常
- [ ] 当意图识别结果格式不正确时，捕获异常
- [ ] 当 `results` 字段缺失或格式错误时，返回友好的错误提示
- [ ] 不会因为数据格式问题导致接口崩溃

### 6. 日志记录

#### 6.1 日志完整性
- [ ] 使用 `@log()` 装饰器记录接口调用
- [ ] 记录用户输入的问题文本
- [ ] 记录意图识别结果（type、similarity、database）
- [ ] 记录生成的推荐问题列表
- [ ] 记录异常信息（如果发生）

#### 6.2 日志格式
- [ ] 日志格式符合项目规范
- [ ] 日志级别设置正确（info、error等）
- [ ] 日志内容清晰，便于调试和监控

### 7. 性能要求

#### 7.1 响应时间
- [ ] 接口响应时间 < 2秒（正常情况）
- [ ] 意图识别服务响应时间 < 1.5秒
- [ ] 问题生成逻辑执行时间 < 0.1秒

#### 7.2 并发处理
- [ ] 支持多用户并发请求
- [ ] 不会因为并发请求导致数据混乱
- [ ] 不会因为并发请求导致性能显著下降

### 8. 响应格式

#### 8.1 成功响应
- [ ] 响应格式符合规范：
```json
{
  "code": 0,
  "message": "成功",
  "data": [
    {"guess_your_question": "问题1"},
    {"guess_your_question": "问题2"},
    {"guess_your_question": "问题3"}
  ]
}
```

#### 8.2 错误响应
- [ ] 错误响应格式符合规范：
```json
{
  "code": 1,
  "message": "错误信息",
  "data": []
}
```

### 9. 集成测试

#### 9.1 与意图识别服务集成
- [ ] 能够正确调用 `IntentService.process_query()` 方法
- [ ] 能够正确解析意图识别结果
- [ ] 能够处理意图识别服务的各种返回情况

#### 9.2 与前端集成
- [ ] 前端能够正常调用接口
- [ ] 前端能够正确解析响应数据
- [ ] 前端能够正确展示推荐问题

### 10. 代码质量

#### 10.1 代码规范
- [ ] 代码符合项目编码规范
- [ ] 函数命名清晰，职责单一
- [ ] 代码注释完整，便于理解

#### 10.2 可维护性
- [ ] 类别1的固定问题易于修改
- [ ] 问题生成逻辑清晰，易于扩展
- [ ] 错误处理逻辑完善，易于调试

## 测试用例

### 测试用例1：类别1推荐问题
**输入**：`{"question": "附近有哪些应急避难所？"}`
**预期输出**：返回3个固定的推荐问题
**验证点**：问题数量、格式、内容

### 测试用例2：类别2推荐问题
**输入**：`{"question": "岱山有多少家化工企业？"}`
**预期输出**：返回第2-4个相关问题
**验证点**：问题来源、数量、格式

### 测试用例3：空问题
**输入**：`{"question": ""}`
**预期输出**：返回错误提示
**验证点**：错误码、错误信息

### 测试用例4：意图识别失败
**输入**：`{"question": "测试问题"}`（模拟意图识别服务不可用）
**预期输出**：返回错误响应
**验证点**：错误处理、日志记录

### 测试用例5：结果不足3个
**输入**：`{"question": "测试问题"}`（模拟results只有2个元素）
**预期输出**：返回实际可用的结果
**验证点**：边界情况处理

## 验收流程

1. **单元测试**：开发者编写单元测试，覆盖所有核心逻辑
2. **集成测试**：测试与意图识别服务的集成
3. **接口测试**：使用 Postman 或类似工具测试接口
4. **前端联调**：与前端团队联调，确保前后端集成正常
5. **性能测试**：测试接口响应时间和并发处理能力
6. **用户验收**：邀请用户试用，收集反馈

## 验收标准总结

- **功能完整性**：所有功能点都已实现且正常工作
- **错误处理**：所有异常情况都有合理的处理和提示
- **日志记录**：日志完整清晰，便于监控和调试
- **性能达标**：响应时间符合要求，支持并发请求
- **代码质量**：代码规范、可维护、易扩展
- **文档完整**：接口文档、测试文档完整准确
