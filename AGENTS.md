# 开发执行规范（精简版）

## 核心原则

**隔离开发、测试驱动、根因修复。**

## 绝对禁令

- **禁止** 在 `main/master` 直接编码
- **禁止** 没有失败测试就写生产代码（TDD 例外需确认）
- **禁止** 多个猜测性修复一起提交
- **禁止** 跳过代码审查直接结束任务

## 必须遵守

### 1) 准备阶段

- **必须** 确认任务类型与相关 SKILL
- **必须** 创建隔离工作区（如 worktree）
- **必须** 验证基线测试可运行

### 2) 开发阶段（每个功能）

- **必须** 先写失败测试（RED）
- **必须** 用最小实现让测试通过（GREEN）
- **必须** 在绿色状态下重构（REFACTOR）
- **必须** 每轮仅做单一目标修改

### 3) 交付阶段

- **必须** 请求代码审查
- **必须** 修复 Critical/Important 问题
- **必须** 运行与改动直接相关的验证

## 调试流程（强制）

1. **根因调查**：稳定复现 + 定位边界
2. **模式对比**：与可工作实现对比差异
3. **假设验证**：一次只验证一个假设
4. **最小修复**：单点修复并回归验证

## TDD 例外（需用户明确许可）

- 一次性原型
- 生成代码
- 纯配置文件

## 验证清单

- [ ] 已隔离开发环境
- [ ] 已执行 RED→GREEN→REFACTOR
- [ ] 已完成代码审查与问题修复
- [ ] 已完成最小回归验证

# 代码质量规范（精简版）

## 核心原则

**可读、可测、可维护、安全默认开启。**

## 绝对禁令

- **禁止** 硬编码密钥或凭证
- **禁止** 忽略错误与异常分支
- **禁止** 跳过输入验证直接入库/执行
- **禁止** 在生产代码保留调试输出
- **禁止** 未授权情况下绕过安全检查

## 必须遵守

### 代码结构

- **必须** 单一职责：函数只做一件事
- **必须** 控制复杂度：避免深层嵌套，优先提前返回
- **必须** 命名清晰：语义化命名优先于注释

### 健壮性

- **必须** 对外部调用做错误处理
- **必须** 对边界输入做校验（类型/格式/范围）
- **必须** 错误信息区分用户提示与技术日志

### 安全基线

- **必须** 使用环境变量管理敏感信息
- **必须** 防护注入/XSS/CSRF 等常见风险（按技术栈实现）
- **必须** 做鉴权与授权校验（涉及受保护资源时）

### 测试要求

- **必须** 为改动提供对应测试或验证步骤
- **必须** 覆盖关键路径与异常路径

## 推荐做法

- **应该** 小步提交，降低回归范围
- **应该** 提取重复逻辑，避免复制粘贴
- **应该** 使用项目既有格式化与风格规范

## 提交前检查清单

- [ ] 无硬编码密钥
- [ ] 输入已验证，错误已处理
- [ ] 安全边界检查已覆盖
- [ ] 相关测试/验证已通过
- [ ] 变更说明清晰可追踪

# 上下文工程规范（精简版）

## 核心原则

**文档与代码同等重要；目录必须稳定、可检索、可归档。**

## 目录约定

```text
docs/
├── static/      # 稳定文档：架构/设计/API/指南
├── contexts/    # 活跃需求上下文（含索引）
└── archive/     # 已归档需求（只读）
```

## 绝对禁令

- **禁止** 将活跃需求直接写入 `docs/archive/`
- **禁止** 手动跳过 `docs/contexts/.contexts-index.json` 更新
- **禁止** 多个需求混用同一上下文目录
- **禁止** 测试文件长期散落在项目根目录

## 必须遵守

### 文档流程

- **必须** 新需求先建立 `docs/contexts/<context-id>/`
- **必须** 同步更新 `docs/contexts/.contexts-index.json`
- **必须** 需求完成后归档到 `docs/archive/`

### 代码组织

- **必须** 保持 `src/`、`test/`、`scripts/` 分离
- **必须** 模块按功能域组织，避免循环依赖
- **必须** 测试完成后归档到 `test/YYYY-MM-DD_<task>/`

## 推荐做法

- **应该** 为每个上下文至少维护：需求、设计、实施记录
- **应该** 在 `test/README.md` 维护测试归档索引
- **应该** 归档内容设为只读，后续修改走新版本目录

## 标准流程

1. 需求确认
2. 创建上下文目录并登记索引
3. 输出设计与实施文档
4. 编码与测试
5. 归档上下文与测试材料

## 验证清单

- [ ] 文档位于正确目录（`static/contexts/archive`）
- [ ] 上下文索引已更新
- [ ] 测试文件已归档
- [ ] 根目录保持整洁

# 工作流程规范（精简版）

## 核心原则

**先流程后执行，先验证后结论。**

## 绝对禁令

- **禁止** 跳过 SKILL 检查直接开工
- **禁止** 未验证就声称“已完成”
- **禁止** 在主分支直接开发
- **禁止** 测试前先写生产代码（非明确例外）
- **禁止** 未做根因调查就盲目修复
- **禁止** 对可并行任务串行执行

## 必须遵守

- **必须** 在任务开始前判断并调用相关 SKILL（1% 规则）
- **必须** 多步骤任务维护 `update_plan`
- **必须** 每批改动后执行最小可证明验证
- **必须** 阻塞时立即停止并说明阻塞点
- **必须** 在响应中给出“已做什么 + 如何验证”

## 标准流程

1. 识别任务类型与目标
2. 检查并调用相关 SKILL
3. 制定计划并标记进行中步骤
4. 小步执行并持续验证
5. 汇总结果与风险
6. 请求反馈后进入下一批

## 阻塞处理

出现以下任一情况时，停止继续实现并先汇报：

- 缺少依赖或权限
- 测试持续失败且原因不明
- 需求存在歧义
- 连续 3 次尝试无有效进展

## 验证清单

- [ ] 已检查/调用相关 SKILL
- [ ] 已维护 `update_plan`
- [ ] 已执行实际验证命令
- [ ] 已记录验证结果与结论
- [ ] 阻塞时已明确说明原因与下一步
