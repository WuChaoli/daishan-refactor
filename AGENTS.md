# 工作流程规范（精简版）

## 核心原则

**先流程后执行，先验证后结论。**

## 绝对禁令

- **禁止** 跳过 SKILL 检查直接开工。
- **禁止** 未验证就声称“已完成”。
- **禁止** 在主分支直接开发。
- **禁止** 测试前先写生产代码（非明确例外）。
- **禁止** 未做根因调查就盲目修复。
- **禁止** 对可并行任务串行执行。

## 必须遵守

- **必须** 在任务开始前判断并调用相关 SKILL（1% 规则）。
- **必须** 多步骤任务维护 `update_plan`。
- **必须** 每批改动后执行最小可证明验证。
- **必须** 阻塞时立即停止并说明阻塞点。
- **必须** 在响应中给出“已做什么 + 如何验证”。

## 标准流程

1. 识别任务类型与目标
2. 检查并调用相关 SKILL
3. 制定计划并标记进行中步骤
4. 小步执行并持续验证
5. 汇总结果与风险
6. 请求反馈后进入下一批

## 阻塞处理

出现以下任一情况时，停止继续实现并先汇报：

- 缺少依赖或权限
- 测试持续失败且原因不明
- 需求存在歧义
- 连续 3 次尝试无有效进展

- [ ] 已维护 `update_plan`
- [ ] 已执行实际验证命令

- [ ] 阻塞时已明确说明原因与下一步

# 代码质量规范（精简版）

## 核心原则

**可读、可测、可维护、安全默认开启。**

## 绝对禁令

- **禁止** 硬编码密钥或凭证。
- **禁止** 忽略错误与异常分支。
- **禁止** 跳过输入验证直接入库/执行。
- **禁止** 在生产代码保留调试输出。
- **禁止** 未授权情况下绕过安全检查。

## 必须遵守

### 代码结构

- **必须** 单一职责：函数只做一件事。
- **必须** 控制复杂度：避免深层嵌套，优先提前返回。
- **必须** 命名清晰：语义化命名优先于注释。

### 健壮性

- **必须** 对外部调用做错误处理。
- **必须** 对边界输入做校验（类型/格式/范围）。
- **必须** 错误信息区分用户提示与技术日志。

### 安全基线

- **必须** 使用环境变量管理敏感信息。
- **必须** 防护注入/XSS/CSRF 等常见风险（按技术栈实现）。
- **必须** 做鉴权与授权校验（涉及受保护资源时）。

### 测试要求

- **必须** 为改动提供对应测试或验证步骤。
- **必须** 覆盖关键路径与异常路径。

## 推荐做法

- **应该** 小步提交，降低回归范围。
- **应该** 提取重复逻辑，避免复制粘贴。
- **应该** 使用项目既有格式化与风格规范。

## Python 开发规范

### 环境与依赖

| 规则 | 命令 |
|------|------|
| 强制 uv（禁用 pip） | `uv add requests` / `uv add --dev pytest` |
| 虚拟环境 `.venv/` | `uv venv --python 3.11` |
| 锁文件提交 | `uv sync --frozen` |

### 数据建模

| 原则 | 要求 |
|------|------|
| 优先类包装 | 跨函数/模块传递必须用类，禁止裸字典/元组 |
| 类型即文档 | 所有字段必须类型注解 |
| 行为内聚 | 相关方法放在类内部 |
| 单文件单类 | 1 个公开类 + N 个私有函数，文件名 = 类名蛇形 |

### 错误处理（快速失败）

| 层级 | 策略 |
|------|------|
| 内部函数 | 直接抛出异常 `if not x: raise ValueError(msg)` |
| 端点函数 | 捕获并返回 `{"code":1, "message":"", "data":None}` |
| 异常类型 | 捕获具体异常（`ValueError`/`KeyError`），避免 `Exception` |
| 日志记录 | 业务错误 `logger.error(msg)`，系统异常 `exc_info=True` |
