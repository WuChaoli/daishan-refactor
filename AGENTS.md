## 交互规则

- 当用户下达命令的时候，主动向用户进行提问，把所有不明确的地方都进行确认，再开始任务执行。
- 并行提问所有不依赖的问题，减少交互成本
- 苏格拉底式提问，直到需求任务清晰为止

**禁止**默认理解用户的需求并开始执行

# 上下文工程规范

## 核心原则

**文档与代码同等重要；目录必须稳定、可检索、可归档。**

## 目录约定

```text
docs/
├── static/      # 稳定文档：架构/设计/API/指南
├── contexts/    # 活跃需求上下文（含索引）
└── archive/     # 已归档需求（只读）
```

## 绝对禁令

- **禁止** 将活跃需求直接写入 `docs/archive/`
- **禁止** 手动跳过 `docs/contexts/.contexts-index.json` 更新
- **禁止** 多个需求混用同一上下文目录
- **禁止** 测试文件长期散落在项目根目录

## 必须遵守

### 文档流程

- **必须** 新需求先建立 `docs/contexts/<context-id>/`
- **必须** 同步更新 `docs/contexts/.contexts-index.json`
- **必须** 需求完成后归档到 `docs/archive/`

### 代码组织

- **必须** 保持 `src/`、`test/`、`scripts/` 分离
- **必须** 模块按功能域组织，避免循环依赖
- **必须** 测试完成后归档到 `test/YYYY-MM-DD_<task>/`

## 推荐做法

- **应该** 为每个上下文至少维护：需求、设计、实施记录
- **应该** 在 `test/README.md` 维护测试归档索引
- **应该** 归档内容设为只读，后续修改走新版本目录

## 标准流程

1. 需求确认
2. 创建上下文目录并登记索引
3. 输出设计与实施文档
4. 编码与测试
5. 归档上下文与测试材料

## 验证清单

- [ ] 文档位于正确目录（`static/contexts/archive`）
- [ ] 上下文索引已更新
- [ ] 测试文件已归档
- [ ] 根目录保持整洁
